On 2026-04-14, the trading engine was briefly deployed with LIVE_TRADING_ENABLED=true due to a hardcoded value in the CI/CD deploy template. Two strategies — v4_fusion and v4_down_only — placed real on-chain orders on Polymarket. Additionally, the margin engine (eu-west-2, Hyperliquid/Binance) was running simultaneously with no coordinated risk gate, placing 2 BTC long perps at ~$75,205 with 5x leverage ($24.97 collateral each).

The CLOB (order book feed) had stale pricing: the engine priced NO tokens at $0.18–$0.20 when the live Polymarket market showed $0.43 — a 139% divergence. Despite this, the signal direction was correct (bearish), and the stale-priced trades won by luck. The CLOB reconciler later caught and logged the divergence. Paper mode was re-engaged by 14:42 UTC, and all three lock layers (.env, DB system_state, CI/CD template) are now confirmed false.

Risk verdict: The incident was survivable — net Polymarket P&L on resolved trades was +$9.83 — but the wallet drawdown reflects open positions placed at mispriced entries that have not yet resolved, plus capital deployed to margin. Had the signal direction been wrong, this would have been a catastrophic loss given the CLOB price divergence magnitude.

The signal quality — even at low confidence bands (0.13–0.20) — held directional integrity. Entry reason labels confirm appropriate regime classification (volatile_trend_down, chop_down).

CLOB Reconciler Caught and Logged the Divergence
The 30-second staleness guard added in a previous session fired and logged the CLOB price divergence. The reconciler correctly identified that CLOB feed data was stale (>30s since last update) and flagged the mismatch between feed prices ($0.18–0.20) and live book ($0.43). Without this guard, the engine would have silently traded on arbitrarily old prices.
CLOB staleness guard threshold: 30s  |  Logged and alerted via Telegram
Paper Mode Successfully Re-Engaged After Detection
Within approximately 5–8 minutes of the final LIVE trade (4484, resolved 14:37 UTC), paper mode was re-locked across all three layers. Trade 4485 (14:42 UTC) shows a paper-* order ID, confirming mode switch was immediate. The three-layer lock (env + DB + CI/CD) is now the correct architecture.
Switch latency: ~5 min from detection to lock

5 What Failed
CRITICAL  CI/CD Template Had LIVE_TRADING_ENABLED=true Hardcoded
The Railway/GitHub CI/CD deploy template contained LIVE_TRADING_ENABLED=true as a hardcoded environment variable. Every engine redeploy overwrote the Montreal .env setting with this value. This is the root cause of the entire incident — a single misconfigured template propagated live trading mode to production without any human confirmation step.
Root cause: Hardcoded env var in CI/CD template  |  Fix: Template now set to false (confirmed)
CRITICAL  CLOB Stale Prices — Engine Traded at 139% Divergence From Market
The CLOB order book feed (ground truth Polymarket bid/ask) became stale, causing the engine to price NO tokens at $0.18–$0.20 when the live market was at $0.43. The engine executed real trades based on these stale prices without a hard validation gate to reject orders when divergence exceeds a threshold. The 30s staleness guard logs the issue but does not block order execution. Winning on a 139% mispriced entry is survivorship bias, not validation.
Max divergence: $0.18 vs $0.43 = 139%  |  Stale price implication: 82–90% implied NO probability vs true ~57%  |  Fix needed: Hard block on CLOB staleness before order placement
HIGH  v10_gate Strategy Incorrectly Set to LIVE Mode
v10_gate was operating in LIVE mode when it should have been GHOST. This resulted in 2 real trades (losses of $0.92 and $1.03) being placed via the v10 DUNE_TRANSITION signal. The v10 strategy had been explicitly set to GHOST in commit 0ae3d16 ("v10_gate mode LIVE→GHOST — only v4_fusion should be LIVE"), but the deploy overrode this.
V10 losses: -$1.95 total  |  Fix: v10 → GHOST (now confirmed)
HIGH  Margin Engine Running Simultaneously Without Coordinated Risk Gate
The margin engine (eu-west-2, Hyperliquid/Binance) placed 2x LONG BTC perp trades at ~$75,205 with 5x leverage ($24.97 collateral each) while the Polymarket engine was also placing live trades. There is no shared risk budget, no position limit coordination, and no single kill switch that halts both engines simultaneously. The Hub margin API returned network errors at audit time, indicating the margin engine may be offline or unreachable.
Capital at risk: ~$250 notional per trade (5x leverage)  |  Coordinated kill: Not implemented
HIGH  56% Wallet Drawdown ($57 → $25.27 USDC)
The wallet dropped from ~$57 to $25.27 USDC over the course of the incident period. The drawdown is largely attributable to open positions (not yet resolved) entered at mispriced levels plus capital deployed to margin. With 12 open LIVE positions holding ~$31.73 in exposure, the outcome remains uncertain. The wallet is now below the $30 balance flagged in the system status baseline.
Open LIVE exposure: ~$31.73  |  Realized loss today: -$4.55  |  If all open positions resolve as losses: wallet → ~$0
MEDIUM  DB paper_mode State vs .env — Two Sources of Truth
The engine reads paper/live mode from both the .env file and the DB system_state table. These can diverge: CI/CD deploys overwrite .env, while the DB retains its last set value. The v58 wallet status API reports the DB state (paper_enabled: true, live_enabled: false) independently of what .env actually says at the Montreal instance. There is no startup check that asserts both agree before trading begins.
Risk: Silent mode mismatch on every deploy  |  Fix: Single source of truth + startup assertion
MEDIUM  Position Duplication — 4 Orders Per Signal (11:57 UTC)
At 11:57 UTC, a single bearish signal on window 1776167700 triggered 4 independent LIVE fill attempts (IDs 4469–4472), each placing $3.03 into the same 5-minute window at slightly different entry prices ($0.57–$0.61). This is consistent with a position de-duplication failure or a missed idempotency check in the order manager. Total capital deployed in one window: $12.12 (vs intended single-stake sizing).
Duplicate stake: 4x $3.03 = $12.12 per window (4x intended)
MEDIUM  No Pre-Trade Checklist Gate Before Live Session
There is no formal pre-trade checklist that must be manually cleared before live trading is enabled. Mode transitions from paper to live happen via env var or DB toggle without requiring explicit sign-off on: CLOB feed freshness, VPIN thresholds, wallet balance sufficiency, or margin engine coordination status.
6 Fix List
1
CLOB entry price validation gate before order placement OPEN
Reject any order where |clob_price - live_book_price| / live_book_price > 0.15 (15% max divergence). Log and skip (do not paper-fallback silently). Alert via Telegram with divergence magnitude. Requires cross-referencing CLOB feed age against a freshness timestamp.
2
Single source of truth for paper/live mode — DB-authoritative startup check OPEN
Engine startup must read mode from DB only. .env LIVE_TRADING_ENABLED should be deprecated or treated as a safety override only (always defaults to false on fresh deploy). Add assertion at startup: if DB says paper=true, engine must operate in paper mode regardless of .env.
3

v10_gate strategy mode LIVE → GHOST FIXED
Fixed in commit 0ae3d16. v10_gate now GHOST. Only v4_fusion authorized for LIVE. Verified via DB strategy config query and system_state check.
5
Coordinated mode switching — single kill switch for all engines OPEN
The Hub /api/system/kill endpoint must halt both Polymarket engine and margin engine simultaneously. Currently only Polymarket engine responds. Margin engine (eu-west-2) has no shared state with Hub. Requires: margin engine to poll Hub kill-switch state on 10s interval, or shared Redis/DB kill flag both engines watch.
6
Position de-duplication — max 1 fill per window per strategy OPEN
Order manager must check for existing open positions in the same market window before placing a new order. If open_positions.count(window_ts, strategy) > 0, skip. The 11:57 UTC burst (4x $3.03 in one window) demonstrates this check is absent or broken.
7
Pre-trade checklist gate before enabling LIVE OPEN
Add a /api/system/live-preflight endpoint that must return 200 before /api/system/resume or any live-enable toggle is accepted. Checklist: CLOB feed fresh (<30s), VPIN feed active, wallet balance > minimum, margin engine status known, DB mode = paper confirmed.
8
Margin engine Hub integration — status + kill switch IN PROGRESS
Margin API endpoints at Hub return {"detail":"Network error: "} — the margin engine is either offline or its Hub connection is broken. This must be repaired before any next live session. Target: margin positions and status visible in Hub dashboard, margin engine respects Hub kill switch.
