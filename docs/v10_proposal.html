Why 2s Polling?

v9: 10s polling (19 evals)

• Chainlink updates every ~30s
• Tiingo updates every 2s
• Miss fast agreement windows
• CL+TI may agree for only 4-6s then diverge
• 10s gap = can miss the entire agreement

v10: 2s polling (90 evals)

• Catch every Tiingo update (2s aligned)
• DUNE queried every 2s with fresh data
• Capture brief agreement windows
• DUNE P changes with each price tick
• More opportunities to catch high-confidence moments
DB impact: ~90 signal_eval writes per window. At 12 windows/hour = ~1,080 writes/hour. Manageable.

Implementation

Phase 1: Shadow Mode (48h)
Log DUNE prediction alongside every v9 signal evaluation. Don't change trading behavior.
Validate: does DUNE P ≥ 0.65 predict oracle correctly at ≥80% rate?

Phase 2: DUNE Gate (when validated)
Add DUNE query to eval pipeline. Only trade when CL+TI agree AND DUNE P ≥ 0.65.
Dynamic cap = DUNE_P - 0.05. All behind V10_DUNE_ENABLED feature flag.

Phase 3: 2s Polling
Reduce eval interval from 10s to 2s. More opportunities to catch agreement+confidence.
FIVE_MIN_EVAL_INTERVAL=2 (new env var, currently implicit from offset list).
